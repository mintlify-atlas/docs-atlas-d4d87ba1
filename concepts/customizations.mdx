## Overview

EUW extends multiple standard ERPNext DocTypes without modifying core files. This is accomplished through Frappe's Custom Field system and the fixtures mechanism, which allows for version-controlled, reproducible customizations.

## Customization Strategy

EUW follows Frappe best practices for extending standard functionality:

<Steps>
  <Step title="Custom Fields">
    Add new fields to existing DocTypes using Custom Field definitions
  </Step>
  <Step title="Property Setters">
    Modify properties of existing fields (visibility, requirements, etc.)
  </Step>
  <Step title="Fixtures Export">
    Store all customizations as JSON fixtures in version control
  </Step>
  <Step title="Automatic Import">
    Frappe automatically applies fixtures during installation and updates
  </Step>
</Steps>

## Fixtures System

### Configuration

EUW declares its fixtures in hooks.py:

```python hooks.py
fixtures = [
    'Client Script',
    'Custom Field',
    'Property Setter',
    'Print Format',
    'Translation'
]
```

### Fixture Types

<AccordionGroup>
  <Accordion title="Custom Field" icon="plus">
    Adds new fields to existing DocTypes without modifying the original schema.
    
    **Location**: `fixtures/custom_field.json`
    
    **Purpose**: Extend standard DocTypes with domain-specific fields
  </Accordion>
  
  <Accordion title="Property Setter" icon="wrench">
    Modifies properties of existing fields (hidden, required, default values, etc.).
    
    **Location**: `fixtures/property_setter.json`
    
    **Purpose**: Customize behavior of standard fields
  </Accordion>
  
  <Accordion title="Client Script" icon="code">
    JavaScript code that runs in the browser to customize form behavior.
    
    **Location**: `fixtures/client_script.json`
    
    **Purpose**: Add client-side validation, auto-fill logic, and UI enhancements
  </Accordion>
  
  <Accordion title="Print Format" icon="print">
    Custom print layouts for documents.
    
    **Location**: `fixtures/print_format.json`
    
    **Purpose**: Branded, formatted document printing
  </Accordion>
  
  <Accordion title="Translation" icon="language">
    Custom translations for labels and messages.
    
    **Location**: `fixtures/translation.json`
    
    **Purpose**: Internationalization support
  </Accordion>
</AccordionGroup>

## Extended DocTypes

EUW adds custom fields to multiple standard ERPNext DocTypes:

### Item

**Purpose**: Link items to applicability categories

```json euw/custom/item.json
{
  "custom_fields": [
    {
      "dt": "Item",
      "fieldname": "custom_applicability",
      "fieldtype": "Link",
      "options": "Applicability",
      "label": "Applicability",
      "insert_after": "stock_uom"
    }
  ]
}
```

**Use case**: Tag items with their applicability (e.g., which aircraft models they're compatible with)

### Asset

## Property Setters

Property Setters modify existing field behavior without adding new fields.

### Example: Item DocType Modifications

```json fixtures/property_setter.json
[
  {
    "doc_type": "Item",
    "field_name": "naming_series",
    "property": "hidden",
    "value": "1"
  },
  {
    "doc_type": "Item",
    "field_name": "naming_series",
    "property": "reqd",
    "value": "0"
  },
  {
    "doc_type": "Item",
    "field_name": "item_code",
    "property": "hidden",
    "value": "0"
  },
  {
    "doc_type": "Item",
    "field_name": "item_code",
    "property": "reqd",
    "value": "1"
  }
]
```

<Info>
This configuration:
- Hides the naming series field
- Makes naming series optional
- Shows the item code field
- Makes item code required

Effectively switching from series-based to manual item code entry.
</Info>

## Managing Customizations

### Exporting Fixtures

To export customizations to fixtures:

```bash
bench --site [sitename] export-fixtures
```

This updates all JSON files in the `fixtures/` directory.

### Importing Fixtures

Fixtures are automatically imported during:

```bash
# Initial installation
bench --site [sitename] install-app euw

# Manual import
bench --site [sitename] import-fixtures euw
```

### Viewing Custom Fields in UI

<Steps>
  <Step title="Navigate to DocType">
    Setup > Customize > [DocType Name] (e.g., "Item")
  </Step>
  <Step title="View Custom Fields">
    Custom fields appear with a "Custom" badge
  </Step>
  <Step title="Edit if Needed">
    Changes made in UI can be exported back to fixtures
  </Step>
</Steps>

<Warning>
Always export customizations after making changes in the UI to keep fixtures in sync.
</Warning>

## Custom Field Naming Convention

EUW follows two naming patterns:

1. **System-generated fields**: Prefixed with `custom_`
   ```json
   {"fieldname": "custom_applicability"}
   ```

2. **Explicit names**: For HR module fields (no prefix)
   ```json
   {"fieldname": "employment_type"}
   ```

<Tip>
Using the `custom_` prefix helps distinguish between standard and custom fields, especially when troubleshooting.
</Tip>

## Field Insertion Strategy

Custom fields use the `insert_after` property to control placement:

```json
{
  "fieldname": "custom_applicability",
  "insert_after": "stock_uom"
}
```

This ensures:
- Logical field grouping
- Consistent form layout
- Compatibility with future ERPNext updates

## Conditional Display

Some fields use `depends_on` for conditional visibility:

```json
{
  "fieldname": "health_insurance_no",
  "depends_on": "eval:doc.health_insurance_provider"
}
```

**Result**: The insurance number field only appears when a provider is selected.

## Fetch From Logic

Automatic field population from linked documents:

```json
{
  "fieldname": "payroll_cost_center",
  "fetch_from": "department.payroll_cost_center",
  "fetch_if_empty": 1
}
```

**Behavior**:
1. When department is selected on Employee
2. If payroll_cost_center is empty
3. Automatically fetch from department.payroll_cost_center

## Troubleshooting

<AccordionGroup>
  <Accordion title="Custom fields not appearing">
    **Solutions**:
    ```bash
    # Re-import fixtures
    bench --site [sitename] import-fixtures euw
    
    # Clear cache
    bench --site [sitename] clear-cache
    
    # Reload DocType
    bench --site [sitename] reload-doc euw [doctype-name]
    ```
  </Accordion>
  
  <Accordion title="Property setter not applying">
    **Solutions**:
    - Check property name spelling
    - Verify field name matches exactly
    - Property setters apply after custom fields
    - Try deleting and re-importing
  </Accordion>
  
  <Accordion title="Fixture import errors">
    **Solutions**:
    - Validate JSON syntax
    - Check for duplicate fieldnames
    - Ensure referenced DocTypes exist
    - Review error logs: `bench --site [sitename] console`
  </Accordion>
</AccordionGroup>

## Migration Considerations

When upgrading ERPNext:

<Warning>
1. **Backup first**: Always backup before major upgrades
2. **Test in staging**: Verify customizations work with new version
3. **Check conflicts**: Review ERPNext changelog for field name conflicts
4. **Update fixtures**: Re-export fixtures after successful upgrade
</Warning>
